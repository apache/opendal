18//! The core concepts of OpenDAL's public API.
19//!
20//! OpenDAL provides a unified abstraction for accessing storage services.
21//!
22//! Public users mostly work with three concepts:
23//!
24//! - [`Builder`]: configures and builds a storage service.
25//! - [`Operator`]: owns a service stack and exposes storage operations such as
26//!   `read`, `write`, and `list`.
27//! - [`Layer`][crate::raw::Layer]: wraps an operator's service stack or runtime
28//!   resources to add behavior such as retry, timeout, tracing, and metrics.
29//!
30//! If you are interested in internal implementation details, please have a look at [`internals`][super::internals].
31//!
32//! # Builder
33//!
34//! A [`Builder`] configures and creates the underlying storage service. Service
35//! crates expose builders through [`services`][crate::services], for example
36//! [`services::Memory`][crate::services::Memory].
37//!
38//! ```text
39//! ┌─────────┐   build()   ┌─────────┐
40//! │ Builder ├────────────►│ Service │
41//! └─────────┘             └─────────┘
42//! ```
43//!
44//! ```no_run
45//! use opendal_core::services::Memory;
46//!
47//! let builder = Memory::default();
48//! ```
49//!
50//! # Operator
51//!
52//! An [`Operator`] is the public handle for a storage service stack. It stores:
53//!
54//! - a base service created from the builder.
55//! - a base [`OperationContext`] with runtime resources such as HTTP transport
56//!   and executor.
57//! - an ordered list of layers.
58//! - the composed service and context used by storage operations.
59//!
60//! ```text
61//! ┌─────────┐   Operator::new   ┌────────────────────────────┐
62//! │ Builder ├──────────────────►│ Operator                   │
63//! └─────────┘                   │ - base service             │
64//!                               │ - base OperationContext    │
65//!                               │ - layers                   │
66//!                               │ - composed service/context │
67//!                               └────────────────────────────┘
68//! ```
69//!
70//! `Operator::new` returns a ready-to-use operator. There is no separate
71//! `finish` step.
72//!
73//! ```no_run
74//! # use opendal_core::Result;
75//! use opendal_core::services::Memory;
76//! use opendal_core::Operator;
77//!
78//! # fn test() -> Result<()> {
79//! let builder = Memory::default();
80//!
81//! let op = Operator::new(builder)?;
82//! # Ok(())
83//! # }
84//! ```
85//!
86//! - `Operator` is cheap to clone because it stores shared handles internally.
87//! - `Operator` has no generic parameters or lifetimes, so it is easy to pass
88//!   through application code.
89//! - `Operator` is `Send` and `Sync`, so it can be shared across threads.
90//! - Methods that change layers or runtime resources return a new operator;
91//!   existing clones and in-flight operations keep using their current composed
92//!   service and context.
93//!
94//! # Runtime resources and layers
95//!
96//! [`OperationContext`] carries runtime resources from the operator to services,
97//! such as [`HttpTransporter`][crate::HttpTransporter] and [`Executor`][crate::Executor].
98//! Operation arguments such as ranges, versions, and concurrency limits stay in
99//! the `Op*` argument structs used by each operation.
100//!
101//! Use [`Operator::with_context`] to replace the base context. Use
102//! [`Operator::layer`] to append a layer. In both cases, OpenDAL replays the full
103//! layer list from the base service and base context to produce a fresh composed
104//! service and context.
105//!
106//! ```text
107//! base service ─┐
108//!               ├─ layer replay ─► composed service ─┐
109//! base context ─┘                                     ├─► operation dispatch
110//!                                                     │    srv.read(&ctx, ...)
111//!                                                     ▼
112//!                                             composed context
113//! ```
114//!
115//! ```no_run
116//! # use opendal_core::Result;
117//! use opendal_core::HttpTransporter;
118//! use opendal_core::OperationContext;
119//! use opendal_core::Operator;
120//! use opendal_core::services::Memory;
121//!
122//! # fn test() -> Result<()> {
123//! let transport = HttpTransporter::default();
124//! let op = Operator::new(Memory::default())?.with_context(
125//!     OperationContext::new().with_http_transport(transport),
126//! );
127//! # let _ = op;
128//! # Ok(())
129//! # }
130//! ```
131//!
132//! # Operations
133//!
134//! After creating an operator, use it to run operations on normalized paths.
135//!
136//! ```text
137//!                            ┌──────────────┐
138//!                  ┌────────►│ read("abc")  │
139//!                  │         └──────────────┘
140//! ┌───────────┐    │
141//! │ Operator  │    │         ┌──────────────┐
142//! │ ┌───────┐ ├────┼────────►│ write("def") │
143//! │ │Service│ │    │         └──────────────┘
144//! └─┴───────┴─┘    │
145//!                  │         ┌──────────────┐
146//!                  └────────►│ list("ghi/") │
147//!                            └──────────────┘
148//! ```
149//!
150//! We can read data with given path in this way:
151//!
152//! ```no_run
153//! # use opendal_core::Result;
154//! use opendal_core::services::Memory;
155//! use opendal_core::Operator;
156//!
157//! # async fn test() -> Result<()> {
158//! let builder = Memory::default();
159//!
160//! let op = Operator::new(builder)?;
161//! let bs: Vec<u8> = op.read("abc").await?;
162//! # Ok(())
163//! # }
164//! ```
165//!
166//! [`Builder`]: crate::Builder
167//! [`Operator`]: crate::Operator
168//! [`OperationContext`]: crate::OperationContext
