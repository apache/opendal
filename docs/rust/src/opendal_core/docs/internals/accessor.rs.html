18//! The internal implementation details of [`Service`].
19//!
20//! [`Service`] is the core trait of OpenDAL's raw API. We operate
21//! underlying storage services via APIs provided by [`Service`].
22//!
23//! # Introduction
24//!
25//! [`Service`] can be split in the following parts:
26//!
27//! ```ignore
28//! //                  <----------Trait Bound-------------->
29//! pub trait Service: Send + Sync + Debug + Unpin + 'static {
30//!     fn info(&self) -> ServiceInfo;
31//!     fn capability(&self) -> Capability;
32//!
33//!     type Reader: oio::Read;
34//!     type Writer: oio::Write;
35//!     type Lister: oio::List;
36//!     type Deleter: oio::Delete;
37//!     type Copier: oio::Copy;
38//!
39//!     fn read(
40//!         &self,
41//!         ctx: &OperationContext,
42//!         path: &str,
43//!         args: OpRead,
44//!     ) -> impl Future<Output = Result<Self::Reader>> + MaybeSend;
45//! }
46//! ```
47//!
48//! Let's go deep into [`Service`] line by line.
49//!
50//! ## Trait Bound
51//!
52//! First we will read the declaration of [`Service`] trait:
53//!
54//! ```ignore
55//! pub trait Service: Send + Sync + Debug + Unpin + 'static {}
56//! ```
57//!
58//! There are many trait boundings here. For now, [`Service`] requires the following bound:
59//!
60//! - [`Send`]: Allow user to send between threads without extra wrapper.
61//! - [`Sync`]: Allow user to sync between threads without extra lock.
62//! - [`Debug`][std::fmt::Debug]: Allow users to print underlying debug information of service.
63//! - [`Unpin`]: Make sure `Service` can be safely moved after being pinned, so users don't need to `Pin<Box<A>>`.
64//! - `'static`: Make sure `Service` is not a short-time reference, allow users to use `Service` in closures and futures without playing with lifetime.
65//!
66//! Implementer of `Service` should take care of the following things:
67//!
68//! - Implement `Debug` for backend, but don't leak credentials.
69//! - Make sure the backend is `Send` and `Sync`, wrap the internal struct with `Arc<Mutex<T>>` if necessary.
70//!
71//! ## Associated Type
72//!
73//! The middle block of [`Service`] is our associated types. We require
74//! implementers to specify concrete operation body types, avoiding extra
75//! dynamic dispatch inside the backend and typed service wrappers.
76//!
77//! [`Service`] has five associated types so far:
78//!
79//! - `Reader`: reader returned by `read` operation.
80//! - `Writer`: writer returned by `write` operation.
81//! - `Lister`: lister returned by `list` operation.
82//! - `Deleter`: deleter returned by `delete` operation.
83//! - `Copier`: copier returned by `copy` operation.
84//!
85//! Backend implementers should take care of the following things:
86//!
87//! - Return concrete operation body types instead of dynamic trait objects like `oio::Reader`.
88//! - Use `()` as the type if the operation is not supported.
89//! - Implement every operation method; unsupported operations should return
90//!   [`ErrorKind::Unsupported`] explicitly.
91//!
92//! Runtime service wrappers that already receive a [`Servicer`] can forward the
93//! erased `oio::*` body types, but the original backend should not erase itself.
94//!
95//! OpenDAL erases these associated types at the [`ServiceDyn`] boundary.
96//! [`Servicer`] is `Arc<dyn ServiceDyn>` and is the handle used by
97//! [`Operator`] and runtime layer composition. This keeps backend and typed
98//! layer implementations concrete, while the composed operator stack can be
99//! stored and cloned through one object-safe service handle.
100//!
101//! ## API Style
102//!
103//! Every API of [`Service`] follows the same style:
104//!
105//! - All APIs have a unique [`Operation`] and [`Capability`]
106//! - All APIs are orthogonal and do not overlap with each other
107//! - Most APIs accept [`OperationContext`], `path` and `OpXxx`, and returns `RpXxx` with concrete operation bodies.
108//! - Most APIs have `async` and `blocking` variants, they share the same semantics but may have different underlying implementations.
109//!
110//! [`OperationContext`] carries operator resources composed by operator layers,
111//! such as HTTP client and executor. Backend implementations should use this
112//! context instead of storing mutable operator resources in [`ServiceInfo`].
113//!
114//! [`Service`] declares immutable identity facts via [`ServiceInfo`] and
115//! operation capability via [`Service::capability`]:
116//!
117//! ```ignore
118//! impl Service for MyBackend {
119//!     type Reader = MyReader;
120//!     type Writer = ();
121//!     type Lister = ();
122//!     type Deleter = ();
123//!     type Copier = ();
124//!
125//!     fn info(&self) -> ServiceInfo {
126//!         ServiceInfo::new(MY_SCHEME, "/", "")
127//!     }
128//!
129//!     fn capability(&self) -> Capability {
130//!         Capability {
131//!             read: true,
132//!             write: true,
133//!             ..Default::default()
134//!         }
135//!     }
136//! }
137//! ```
138//!
139//! Now that you have mastered [`Service`], let's go and implement our own backend!
140//!
141//! # Tutorial
142//!
143//! This tutorial implements a `duck` storage service that sends API
144//! requests to a super-powered duck. Gagaga!
145//!
146//! ## Scheme
147//!
148//! First of all, let's pick a good scheme for our duck service. The
149//! scheme should be unique and easy to understand. Normally we should
150//! use its formal name.
151//!
152//! For example, we will use `s3` for AWS S3 Compatible Storage Service
153//! instead of `aws` or `awss3`. This is because there are many storage
154//! vendors that provide s3-like RESTful APIs, and our s3 service is
155//! implemented to support all of them, not just AWS S3.
156//!
157//! Obviously, we can use `duck` as scheme, let's add a new constant string:
158//!
159//! ```ignore
160//! pub const DUCK_SCHEME: &str = "duck";
161//! ```
162//!
163//! ## Builder
164//!
165//! Then we can implement a builder for the duck service. The [`Builder`]
166//! will provide APIs for users to configure, and they will create an
167//! instance of a particular service.
168//!
169//! Let's create a `backend` mod under `services/duck` directory, and adding the following code.
170//!
171//! ```ignore
172//! use crate::raw::*;
173//! use crate::*;
174//!
175//! /// Duck Storage Service support. Gagaga!
176//! ///
177//! /// # Capabilities
178//! ///
179//! /// This service can be used to:
180//! ///
181//! /// - [x] read
182//! /// - [ ] write
183//! /// - [ ] list
184//! /// - [ ] presign
185//! ///
186//! /// # Configuration
187//! ///
188//! /// - `root`: Set the work dir for backend.
189//! ///
190//! /// ## Via Builder
191//! ///
192//! /// ```no_run
193//! /// use std::sync::Arc;
194//! ///
195//! /// use anyhow::Result;
196//! /// use opendal_core::services::Duck;
197//! /// use opendal_core::Operator;
198//! ///
199//! /// #[tokio::main]
200//! /// async fn main() -> Result<()> {
201//! ///     // Create Duck backend builder.
202//! ///     let mut builder = DuckBuilder::default();
203//! ///     // Set the root for duck, all operations will happen under this root.
204//! ///     //
205//! ///     // NOTE: the root must be absolute path.
206//! ///     builder.root("/path/to/dir");
207//! ///
208//! ///     let op: Operator = Operator::new(builder)?;
209//! ///
210//! ///     Ok(())
211//! /// }
212//! /// ```
213//! #[derive(Default, Debug, Serialize, Deserialize, Clone, PartialEq, Eq)]
214//! #[serde(default)]
215//! #[non_exhaustive]
216//! pub struct DuckConfig {
217//!     pub root: Option<String>,
218//! }
219//!
220//! #[derive(Default, Clone)]
221//! pub struct DuckBuilder {
222//!     config: DuckConfig,
223//! }
224//! ```
225//!
226//! Now let's implement the required APIs for `DuckConfig`:
227//!
228//! ```ignore
229//! impl Configurator for DuckConfig {
230//!     type Builder = DuckBuilder;
231//!
232//!     fn into_builder(self) -> Self::Builder {
233//!         DuckBuilder { config: self }
234//!     }
235//! }
236//! ```
237//!
238//! Note that `DuckBuilder` is part of our public API, so it needs to be
239//! documented. And any changes you make will directly affect users, so
240//! please take it seriously. Otherwise, you will be hunted down by many
241//! angry ducks.
242//!
243//! Then, we can implement required APIs for `DuckBuilder`:
244//!
245//! ```ignore
246//! impl DuckBuilder {
247//!     /// Set root of this backend.
248//!     ///
249//!     /// All operations will happen under this root.
250//!     pub fn root(&mut self, root: &str) -> &mut Self {
251//!         self.config.root = if root.is_empty() {
252//!             None
253//!         } else {
254//!             Some(root.to_string())
255//!         };
256//!
257//!         self
258//!     }
259//! }
260//!
261//! impl Builder for DuckBuilder {
262//!     type Config = DuckConfig;
263//!
264//!     fn build(self) -> Result<impl Service>  {
265//!         debug!("backend build started: {:?}", &self);
266//!
267//!         let root = normalize_root(&self.config.root.clone().unwrap_or_default());
268//!         debug!("backend use root {}", &root);
269//!
270//!         Ok(DuckBackend { root })
271//!     }
272//! }
273//! ```
274//!
275//! `DuckBuilder` is ready now, let's try to play with real ducks!
276//!
277//! ## Backend
278//!
279//! I'm sure you can see it already: `DuckBuilder` will build a
280//! `DuckBackend` that implements [`Service`]. The backend is what we used
281//! to communicate with the super-powered ducks!
282//!
283//! Let's keep adding more code under `backend.rs`:
284//!
285//! ```ignore
286//! /// Duck storage service backend
287//! #[derive(Clone, Debug)]
288//! pub struct DuckBackend {
289//!     root: String,
290//! }
291//!
292//! impl Service for DuckBackend {
293//!     type Reader = DuckReader;
294//!     type Writer = ();
295//!     type Lister = ();
296//!     type Deleter = ();
297//!     type Copier = ();
298//!
299//!     fn info(&self) -> ServiceInfo {
300//!         ServiceInfo::new(DUCK_SCHEME, &self.root, "")
301//!     }
302//!
303//!     fn capability(&self) -> Capability {
304//!         Capability {
305//!             read: true,
306//!             ..Default::default()
307//!         }
308//!     }
309//!
310//!     fn read(&self, ctx: &OperationContext, path: &str, args: OpRead) -> Result<Self::Reader> {
311//!         gagaga!()
312//!     }
313//! }
314//! ```
315//!
316//! Congratulations, we have implemented a [`Service`] that can talk to
317//! Super Power Ducks!
318//!
319//! What!? There are no Super Power Ducks? So sad, but never mind, we have
320//! really powerful storage services [here](https://github.com/apache/opendal/issues/5). Welcome to pick one to implement. I promise you won't
321//! have to `gagaga!()` this time.
